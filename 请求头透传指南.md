# 渠道请求头覆盖与透传配置指南

## 概述

渠道的**请求头覆盖**（Header Override）功能允许你在 new-api 网关向上游 AI 服务商发送请求时，自定义、覆盖或透传 HTTP 请求头。这对于以下场景非常有用：

- 为不同渠道设置不同的 User-Agent
- 透传客户端身份标识（如 `X-User-ID`、`X-Request-ID`）
- 传递追踪信息（如 `X-Trace-ID`、`X-Span-ID`）
- 自定义认证头（如特殊的 API Key 格式）
- 传递来源信息（如 `X-Forwarded-For`、`Referer`）

## 配置格式

请求头覆盖配置是一个 **JSON 对象**，在渠道的高级设置中填写。

## 一、普通请求头设置

### 基本语法

```json
{
  "Header-Name": "header-value"
}
```

### 示例

```json
{
  "User-Agent": "MyGateway/1.0",
  "X-Custom-Header": "custom-value",
  "X-Request-Source": "new-api-gateway"
}
```

上游服务商会收到这些请求头的精确值。

---

## 二、动态占位符

### 2.1 `{api_key}` - 渠道 API 密钥

在配置值中使用 `{api_key}`，会被替换为**当前渠道的 API Key**。

#### 示例

```json
{
  "Authorization": "Bearer {api_key}",
  "X-API-Key": "{api_key}"
}
```

假设渠道的 API Key 是 `sk-abc123`，上游会收到：
- `Authorization: Bearer sk-abc123`
- `X-API-Key: sk-abc123`

#### 适用场景
- 自定义认证头格式
- 某些服务商需要在多个请求头中传递密钥

---

### 2.2 `{client_header:<name>}` - 客户端请求头值

提取客户端发送给 new-api 的指定请求头，并转发给上游。

#### 语法规则

```json
{
  "Upstream-Header-Name": "{client_header:Client-Header-Name}"
}
```

**重要限制：**
- 必须是**完整的值**，不能与其他文本拼接
- ✅ 正确：`"X-User-ID": "{client_header:X-User-ID}"`
- ❌ 错误：`"X-Info": "user={client_header:X-User-ID}"` （不支持拼接）

#### 示例

```json
{
  "X-Original-User-Agent": "{client_header:User-Agent}",
  "X-Request-ID": "{client_header:X-Request-ID}",
  "X-Trace-ID": "{client_header:X-Trace-ID}",
  "X-Client-IP": "{client_header:X-Real-IP}"
}
```

#### 行为说明

| 客户端请求头 | 配置 | 上游收到 |
|------------|------|---------|
| `X-Request-ID: req-123` | `"X-Request-ID": "{client_header:X-Request-ID}"` | `X-Request-ID: req-123` |
| *未发送 X-Request-ID* | `"X-Request-ID": "{client_header:X-Request-ID}"` | *不发送此请求头* |

**注意：** 如果客户端没有发送指定的请求头，则不会向上游发送该请求头（而不是发送空值）。

---

## 三、请求头透传规则

透传规则允许你**批量转发**客户端的请求头给上游，无需逐一枚举。

### 3.1 `"*"` - 透传所有请求头

#### 语法

```json
{
  "*": ""
}
```

**注意：** value 可以是任意值（`""`、`true`、`false` 等），只要 key 存在即可，推荐使用空字符串 `""`。

#### 行为

客户端发送的**几乎所有**请求头都会被转发给上游，但**受到安全限制**（见下文）。

#### 示例

```json
{
  "*": ""
}
```

客户端发送：
```
User-Agent: Mozilla/5.0
X-Request-ID: req-123
X-Custom-Header: custom-value
Authorization: Bearer client-token
```

上游收到：
```
User-Agent: Mozilla/5.0
X-Request-ID: req-123
X-Custom-Header: custom-value
（Authorization 被安全限制阻止）
```

---

### 3.2 正则表达式匹配透传

#### 语法

```json
{
  "re:<regex-pattern>": "",
  "regex:<regex-pattern>": ""
}
```

支持两种前缀：`re:` 和 `regex:`（效果相同）。

#### 正则语法

使用 Go 标准库的正则表达式（[RE2 语法](https://github.com/google/re2/wiki/Syntax)）。

#### 示例

##### 示例 1：透传所有 `X-Trace-` 开头的请求头

```json
{
  "re:^X-Trace-": ""
}
```

客户端发送：
- `X-Trace-ID: trace-123` ✅ 透传
- `X-Trace-Span: span-456` ✅ 透传
- `X-Request-ID: req-789` ❌ 不匹配，不透传

##### 示例 2：透传多种前缀

```json
{
  "regex:^(X-Trace-|X-Span-|X-Request-)": ""
}
```

匹配以 `X-Trace-`、`X-Span-` 或 `X-Request-` 开头的请求头。

##### 示例 3：透传自定义命名空间

```json
{
  "re:^X-MyApp-.*$": ""
}
```

##### 示例 4：同时使用多个正则规则

```json
{
  "re:^X-Trace-": "",
  "regex:^X-Custom-": "",
  "re:^CF-": ""
}
```

## 四、安全限制（不可透传的请求头）

为了防止安全问题和协议错误，以下请求头**永远不会**被透传（即使匹配了 `"*"` 或正则规则）：

### 4.1 RFC 7230 Hop-by-Hop 头
这些请求头只在单次连接中有效，不应跨代理转发：
- `Connection`
- `Keep-Alive`
- `Proxy-Authenticate`
- `Proxy-Authorization`
- `TE`
- `Trailer`
- `Transfer-Encoding`
- `Upgrade`

### 4.2 认证凭证（防止密钥泄露）
- `Authorization`
- `X-API-Key`
- `X-Goog-API-Key`
- `Cookie`

**说明：** 这些请求头不能通过通配符或正则透传，但可以通过**显式配置**覆盖（见下文优先级）。

### 4.3 传输控制头
- `Host`
- `Content-Length`
- `Accept-Encoding`

### 4.4 WebSocket 握手头
- `Sec-WebSocket-Key`
- `Sec-WebSocket-Version`
- `Sec-WebSocket-Extensions`

---

## 五、执行顺序与优先级

### 执行顺序

1. **先应用透传规则**（`"*"` 和 `"re:..."`)
2. **再应用显式覆盖**（普通键值对和占位符）

### 优先级规则

**显式覆盖 > 透传规则**

#### 示例

```json
{
  "*": "",
  "User-Agent": "MyGateway/2.0"
}
```

**结果：**
- 客户端的 `User-Agent: Mozilla/5.0` 被透传
- 但立即被显式配置的 `User-Agent: MyGateway/2.0` 覆盖
- 上游最终收到：`User-Agent: MyGateway/2.0`

### 绕过安全限制

虽然 `Authorization` 不能通过 `"*"` 透传，但可以**显式配置**：

```json
{
  "*": "",
  "Authorization": "{client_header:Authorization}"
}
```

这样可以主动转发客户端的认证头（谨慎使用）。

## 六、完整配置示例

### 示例 1：基础透传 + 自定义认证

```json
{
  "Authorization": "Bearer {api_key}",
  "User-Agent": "MyGateway/1.0",
  "re:^X-Request-": ""
}
```

**效果：**
- 使用渠道密钥认证
- 固定 User-Agent
- 透传所有 `X-Request-*` 请求头（如 `X-Request-ID`）

---

### 示例 2：全局透传 + 部分覆盖

```json
{
  "*": "",
  "User-Agent": "claude-cli/2.1.172 (external, cli)",
  "Authorization": "Bearer {api_key}"
}
```

**效果：**
- 透传客户端所有请求头（除安全限制）
- 强制覆盖 `User-Agent` 和 `Authorization`

---

### 示例 3：传递客户端身份

```json
{
  "Authorization": "Bearer {api_key}",
  "X-Original-User": "{client_header:X-User-ID}",
  "X-Request-ID": "{client_header:X-Request-ID}",
  "X-Forwarded-For": "{client_header:X-Real-IP}"
}
```

**效果：**
- 提取客户端的用户 ID、请求 ID、真实 IP，并转发给上游

---

### 示例 4：追踪链路透传（分布式追踪）

```json
{
  "Authorization": "Bearer {api_key}",
  "regex:^(X-Trace-|X-Span-|X-B3-)": ""
}
```

**效果：**
- 透传 OpenTelemetry、Zipkin、Jaeger 等追踪头
- 匹配 `X-Trace-ID`、`X-Span-ID`、`X-B3-TraceId` 等

---

### 示例 5：多密钥场景（从客户端传递子密钥）

```json
{
  "Authorization": "Bearer {client_header:X-Sub-API-Key}"
}
```

**场景：**
- 客户端在 `X-Sub-API-Key` 中传递具体的子密钥
- new-api 网关提取并转发给上游

## 七、常见问题

### Q1：透传规则的 value 应该填什么？

**A：** 任意值都可以，推荐使用空字符串 `""`。

```json
{
  "*": "",            // ✅ 推荐
  "*": true,          // ✅ 也可以
  "re:^X-": false     // ✅ 也可以
}
```

透传规则只检查 key，value 会被忽略。

---

### Q2：`{server_ip}` 占位符可以用吗？

**A：** **不支持**。目前只支持 `{api_key}` 和 `{client_header:<name>}`。

如果需要传递 IP：
- 固定 IP：直接写死，如 `"X-Real-IP": "1.2.3.4"`
- 客户端 IP：透传客户端请求头，如 `"X-Real-IP": "{client_header:X-Real-IP}"`

---

### Q3：`"*": ""` 会透传客户端的 Authorization 吗？

**A：** **不会**。`Authorization` 在安全限制列表中，不能通过通配符透传。

如果需要转发客户端认证头，必须显式配置：

```json
{
  "Authorization": "{client_header:Authorization}"
}
```

---

### Q4：正则表达式区分大小写吗？

**A：** **区分大小写**。HTTP 请求头名称本身不区分大小写，但正则匹配时按实际字符匹配。

建议使用大小写不敏感的正则：

```json
{
  "re:(?i)^x-trace-": ""
}
```

`(?i)` 开启不区分大小写模式。

---

### Q5：能否拼接占位符？

**A：** **不支持拼接**。

```json
{
  "X-Info": "user={client_header:X-User-ID}"  // ❌ 错误
}
```

只支持完整替换：

```json
{
  "X-User-ID": "{client_header:X-User-ID}"    // ✅ 正确
}
```

---

### Q6：配置无效怎么排查？

1. **检查 JSON 格式是否正确**
   - 使用 JSON 校验工具（如 [jsonlint.com](https://jsonlint.com)）
   
2. **检查是否触发安全限制**
   - 查看请求头名称是否在禁止列表中

3. **检查客户端是否真的发送了该请求头**
   - 使用抓包工具（如 Wireshark、Charles）或浏览器开发者工具

4. **查看日志**
   - new-api 会记录请求头覆盖解析错误

---

### Q7：透传规则和显式配置冲突时如何处理？

**A：** **显式配置优先**。

```json
{
  "*": "",
  "User-Agent": "MyGateway/1.0"
}
```

即使客户端发送了 `User-Agent: Mozilla/5.0`，上游也会收到 `User-Agent: MyGateway/1.0`。

## 八、注意事项

### 8.1 性能考虑

- 正则表达式会在每次请求时执行，但有缓存优化
- 复杂的正则表达式可能轻微影响性能
- 建议优先使用精确匹配或简单正则

### 8.2 安全建议

- **谨慎透传 Authorization**：除非确实需要，否则不要透传客户端的认证头
- **避免透传敏感信息**：如内部 IP、调试信息等
- **使用正则时注意范围**：避免过于宽泛的匹配规则

### 8.3 调试技巧

1. **使用渠道测试功能**：在渠道管理页面测试配置是否生效
2. **查看上游日志**：某些服务商（如 OpenAI）提供请求日志查看功能
3. **使用中间代理**：通过 Charles、mitmproxy 等工具抓包查看实际请求头

## 九、技术实现细节

### 代码位置
- 解析逻辑：`relay/channel/api_request.go` - `processHeaderOverride()`
- 占位符替换：`applyHeaderOverridePlaceholders()`
- 安全限制列表：`passthroughSkipHeaderNamesLower`

### 正则引擎
使用 Go 标准库 `regexp` 包（基于 RE2），不支持回溯和部分 PCRE 特性。

### 缓存机制
正则表达式编译结果会缓存在 `headerPassthroughRegexCache`（`sync.Map`），避免重复编译。
