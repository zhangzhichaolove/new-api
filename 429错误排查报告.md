# 429错误排查报告

## 一、429错误触发机制

项目中有**6种不同的速率限制器**会触发429错误：

### 1. **全局API速率限制** (GlobalAPIRateLimit)
- **触发条件**: 180秒内超过360次请求（每IP）
- **适用范围**: 所有 `/api/*` 路径
- **实现位置**: `middleware/rate-limit.go:97`
- **触发逻辑**: 
  ```
  if (请求数 > 360次) && (时间窗口 < 180秒) {
      返回 429
  }
  ```

### 2. **全局Web速率限制** (GlobalWebRateLimit)
- **触发条件**: 180秒内超过120次请求（每IP）
- **适用范围**: Web前端路由
- **实现位置**: `router/web-router.go`
- **触发逻辑**: 同上，限制数调整为120次/180秒

### 3. **关键操作速率限制** (CriticalRateLimit)
- **触发条件**: 20分钟内超过20次请求（每IP）
- **适用范围**: 
  - 登录/注册
  - 密码重置
  - OAuth绑定
  - 支付请求
  - 渠道密钥查看
  - Token密钥查看
- **实现位置**: 30+个关键端点
- **触发逻辑**: 
  ```
  if (请求数 > 20次) && (时间窗口 < 1200秒) {
      返回 429
  }
  ```

### 4. **模型请求速率限制** (ModelRequestRateLimit) ⭐重点
- **默认状态**: **关闭**（`ModelRequestRateLimitEnabled = false`）
- **触发条件**（启用后）:
  - **总请求限制**: N分钟内总请求数（包括失败请求）
  - **成功请求限制**: N分钟内成功请求数（状态码<400）
  - 支持按用户组配置不同限制
- **适用范围**: 
  - `/v1/*` (OpenAI格式API)
  - `/v1beta/*` (Gemini格式API)
- **实现位置**: `middleware/model-rate-limit.go`
- **关键特性**:
  - 按**用户ID**限流（非IP）
  - 分别统计总请求和成功请求
  - 支持分组差异化限制

### 5. **搜索端点速率限制** (SearchRateLimit)
- **触发条件**: 60秒内超过10次搜索（每用户）
- **适用范围**: 
  - Token搜索: `/api/token/search`
  - 日志搜索: `/api/log/self/search`
- **触发逻辑**: 基于用户ID限流

### 6. **上传/下载速率限制**
- **上传**: 60秒内10次
- **下载**: 60秒内10次
- **适用范围**: 文件上传下载端点

## 二、当前配置值

### 默认配置（common/constants.go + common/init.go）

| 限制器 | 启用状态 | 请求数 | 时间窗口 | 环境变量 |
|--------|---------|--------|----------|---------|
| Global API | ✅ 启用 | 360次 | 180秒 | `GLOBAL_API_RATE_LIMIT`, `GLOBAL_API_RATE_LIMIT_DURATION` |
| Global Web | ✅ 启用 | 120次 | 180秒 | `GLOBAL_WEB_RATE_LIMIT`, `GLOBAL_WEB_RATE_LIMIT_DURATION` |
| Critical | ✅ 启用 | 20次 | 1200秒(20分钟) | `CRITICAL_RATE_LIMIT`, `CRITICAL_RATE_LIMIT_DURATION` |
| Search | ✅ 启用 | 10次 | 60秒 | `SEARCH_RATE_LIMIT`, `SEARCH_RATE_LIMIT_DURATION` |
| Upload | ✅ 启用 | 10次 | 60秒 | 硬编码 |
| Download | ✅ 启用 | 10次 | 60秒 | 硬编码 |
| **Model Request** | ❌ 关闭 | 0/1000次 | 1分钟 | **无环境变量，仅通过管理后台配置** |

### 模型请求限制（setting/rate_limit.go）

```go
ModelRequestRateLimitEnabled = false              // 默认关闭
ModelRequestRateLimitDurationMinutes = 1          // 1分钟窗口
ModelRequestRateLimitCount = 0                    // 总请求数限制（0=不限制）
ModelRequestRateLimitSuccessCount = 1000          // 成功请求数限制
ModelRequestRateLimitGroup = map[string][2]int{}  // 分组配置
```

## 三、频繁429的可能原因

### 情况A: 如果在AI API调用时遇到429

**最可能原因**: 触发了**Global API速率限制**

```
180秒内超过360次请求 → 平均每秒2次
```

**排查方法**:
```bash
# 1. 检查是否频繁调用
grep "429" your_log_file | wc -l

# 2. 查看访问频率
tail -f logs/*.log | grep -E "POST /v1/chat/completions|429"
```

**解决方案**:
```bash
# 方法1: 临时调高限制（环境变量）
export GLOBAL_API_RATE_LIMIT=1000
export GLOBAL_API_RATE_LIMIT_DURATION=180

# 方法2: 彻底关闭（不推荐）
export GLOBAL_API_RATE_LIMIT_ENABLE=false
```

### 情况B: 如果在登录/注册/支付时遇到429

**原因**: 触发了**Critical速率限制**

```
20分钟内超过20次敏感操作
```

**解决方案**:
```bash
export CRITICAL_RATE_LIMIT=100
export CRITICAL_RATE_LIMIT_DURATION=1200
export CRITICAL_RATE_LIMIT_ENABLE=false  # 或直接关闭
```

### 情况C: 如果在模型API调用时遇到429且用户反馈"封号"感觉

**原因**: 可能**已启用ModelRequestRateLimit**（需确认）

**排查方法**:
```sql
-- 检查数据库配置
SELECT * FROM options WHERE key = 'ModelRequestRateLimitEnabled';
SELECT * FROM options WHERE key = 'ModelRequestRateLimitSuccessCount';
```

**解决方案**: 
- 通过管理后台调整或关闭

## 四、如何定位具体是哪个限流器触发

### 方法1: 通过日志标识符

速率限制器在Redis/内存中使用不同的key前缀：

```
rateLimit:GW:{IP}     → Global Web
rateLimit:GA:{IP}     → Global API
rateLimit:CT:{IP}     → Critical
rateLimit:SR:user:{ID} → Search
rateLimit:UP:{IP}     → Upload
rateLimit:DW:{IP}     → Download
rateLimit:MRRL:{UserID} → Model Request (总请求)
rateLimit:MRRLS:{UserID} → Model Request (成功请求)
```

### 方法2: 查看Redis键

```bash
redis-cli KEYS "rateLimit:*" | head -20
redis-cli GET "rateLimit:GA:127.0.0.1"  # 查看具体计数
```

### 方法3: 添加调试日志

在 `middleware/rate-limit.go:56` 和 `middleware/model-rate-limit.go:93` 添加：

```go
fmt.Printf("[429 DEBUG] Rate limit triggered: mark=%s, key=%s, count=%d/%d\n", 
    mark, key, currentCount, maxRequestNum)
```

## 五、推荐配置调整

### 1. 高流量场景（推荐）

```bash
# .env 或环境变量
GLOBAL_API_RATE_LIMIT_ENABLE=true
GLOBAL_API_RATE_LIMIT=1000           # 提高到1000次
GLOBAL_API_RATE_LIMIT_DURATION=180

GLOBAL_WEB_RATE_LIMIT=500            # 提高到500次
GLOBAL_WEB_RATE_LIMIT_DURATION=180

CRITICAL_RATE_LIMIT=50               # 关键操作放宽到50次
CRITICAL_RATE_LIMIT_DURATION=1200
```

### 2. 内网/可信环境

```bash
# 关闭全局限制，仅保留关键操作限制
GLOBAL_API_RATE_LIMIT_ENABLE=false
GLOBAL_WEB_RATE_LIMIT_ENABLE=false
CRITICAL_RATE_LIMIT_ENABLE=true      # 保留登录注册等的保护
```

### 3. 启用按用户限流（推荐用于生产）

通过管理后台启用**ModelRequestRateLimit**，这样可以：
- 按用户ID限流（而非IP）
- 避免共享IP用户互相影响
- 防止单个用户滥用

## 六、代码修改建议

### 建议1: 增加429响应的详细提示

在 `middleware/rate-limit.go:56, 70` 修改：

```go
// 当前代码
c.Status(http.StatusTooManyRequests)
c.Abort()

// 改进后
c.JSON(http.StatusTooManyRequests, gin.H{
    "error": "rate_limit_exceeded",
    "message": fmt.Sprintf("速率限制：%d秒内最多%d次请求", duration, maxRequestNum),
    "retry_after": duration,
})
c.Abort()
```

### 建议2: 添加Retry-After响应头

```go
c.Header("Retry-After", strconv.FormatInt(duration, 10))
c.Status(http.StatusTooManyRequests)
```

## 七、快速诊断命令

```bash
# 1. 查看当前环境变量配置
env | grep RATE_LIMIT

# 2. 实时监控429错误
tail -f logs/*.log | grep "429"

# 3. 统计最近1小时429错误数
grep "429" logs/*.log | grep "$(date +%Y-%m-%d)" | wc -l

# 4. 查看Redis中的限流键
redis-cli --scan --pattern "rateLimit:*"

# 5. 清空所有限流计数（慎用）
redis-cli KEYS "rateLimit:*" | xargs redis-cli DEL
```

## 八、总结

### 最常见的429触发原因排名

1. **Global API限制** (180秒/360次) - 占90%
2. **Critical限制** (20分钟/20次) - 占8%
3. **ModelRequest限制** - 占2%（大部分安装未启用）
